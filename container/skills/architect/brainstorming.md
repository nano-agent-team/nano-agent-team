# Brainstorming

Use this skill when exploring solution approaches before committing to a design.

## Steps

1. **Define the problem clearly** — write a one-sentence problem statement:
   - What needs to be achieved?
   - What constraints exist? (performance, compatibility, security, team conventions)

2. **Generate at least 3 approaches** — quantity before quality:
   - Approach A: [name] — brief description
   - Approach B: [name] — brief description
   - Approach C: [name] — brief description
   - Do not evaluate yet — just generate

3. **Evaluate each approach** — for each, list:
   - Pros: what makes this approach good?
   - Cons: what are the risks or downsides?
   - Complexity: low / medium / high
   - Reversibility: easy to change later / hard to change

4. **Select the best approach** — state clearly:
   - Which approach you choose and why
   - What trade-offs you are accepting
   - Any risks that need mitigation

5. **Validate the choice** — sanity check:
   - Does it fit within existing architecture?
   - Is it consistent with how the rest of the codebase works?
   - Can a developer implement it in a reasonable timeframe?

## Rules

- Never settle on the first idea without considering alternatives
- Prefer simple, reversible solutions over complex, irreversible ones
- If all approaches have high risk, flag it before proceeding
